# 클라이언트 입력값 신뢰 금지 — readonly는 서버 보안이 아니다

## 1. 문제 제기: "readonly로 막았는데 왜 서버에서 또 검증해야 하지?"

게시글 작성 화면에서 `writer`필드를 로그인한 사용자 정보로 자동 입력하고 `readonly`로 수정을 막았습니다. 언뜻 보기엔 안전해 보이지만, 이 방식만으로는 보안상 허점이 존재합니다.

**핵심 과제: 클라이언트에서 막은 것을 서버에서도 반드시 검증해야 하는 이유는 무엇인가?**

## 2. 원인 분석: readonly는 브라우저에서만 작동한다

`readonly` 속성은 브라우저 UI에서만 입력을 막을 뿐, HTTP 요청 자체를 막지는 못합니다. 누구든 개발자 도구나 직접 POST 요청을 통해 `writer` 값을 임의로 변경해서 서버에 전송할 수 있습니다.

```
// 브라우저에서는 수정 불가
<input type="text" name="writer" value="user1" readonly>

// 하지만 개발자 도구나 직접 POST 요청으로 얼마든지 조작 가능
writer=다른사람아이디
```

**즉 클라이언트에서 오는 값은 절대 신뢰하면 안 됩니다.**

## 3. 해결: 서버에서 직접 writer 세팅

폼에서 받은 `writer` 값을 사용하지 않고, `@AuthenticationPrincipal`로 현재 로그인한 사용자 정보를 직접 받아서 서버에서 세팅했습니다.

```java
@PreAuthorize("isAuthenticated()")
@PostMapping("/write")
public String write(@AuthenticationPrincipal AccountDTO accountDTO,
                    BoardDTO boardDTO, ...) {

    boardDTO.setWriter(accountDTO.getUsername()); // 서버에서 직접 세팅

    boardService.writeBoard(boardDTO, files);
    ...
}
```

이렇게 하면 클라이언트가 `writer` 값을 어떻게 조작하든 서버에서 덮어씌우기 때문에 안전합니다.

## 4. 결론 및 회고

- **클라이언트 신뢰 금지**: `readonly`, `hidden`, `disabled` 등 클라이언트 측 제한은 UX 목적일 뿐, 보안 수단이 될 수 없습니다. 민감한 값은 반드시 서버에서 직접 세팅하거나 검증해야 합니다.
- **`@AuthenticationPrincipal` 활용**: 시큐리티가 관리하는 인증 정보를 컨트롤러에서 직접 받아 활용하면 클라이언트 조작으로부터 안전하게 처리할 수 있습니다.
