# 물리 파일 삭제 실패 시 예외를 삼킬지 던질지 결정

## 1. 문제 제기: "물리 파일 삭제 실패 시 DB도 롤백해야 할까?"

파일 삭제가 포함된 여러 서비스 메서드를 구현하면서, 물리 파일 삭제 실패 시 예외를 던져서 DB도 롤백할지, 아니면 예외를 삼키고 진행할지 결정해야 했습니다.

## 2. 원인 분석

### ① DB와 물리 파일은 완벽한 정합성 보장이 불가능하다

`@Transactional`은 DB 작업만 롤백합니다. 물리 파일 삭제는 파일 시스템에 대한 작업이라 트랜잭션 범위 밖입니다. 어느 순서로 처리하든 둘 중 하나가 실패하면 불일치가 발생합니다.

| 순서             | 실패 케이스      | 결과                           |
| ---------------- | ---------------- | ------------------------------ |
| 물리 삭제 먼저   | DB 업데이트 실패 | 파일은 없는데 DB엔 파일명 남음 |
| DB 업데이트 먼저 | 물리 삭제 실패   | DB는 정상인데 파일이 남음      |

DB 업데이트를 먼저 하는 쪽이 그나마 덜 위험합니다. 유령 파일은 나중에 `@Scheduled`로 정리 가능하지만, DB에 파일명이 남아있으면 사용자에게 직접 영향을 줍니다.

### ② `deleteFiles()`가 예외를 삼키고 있었다

```java
// 기존 코드 - 예외를 삼키고 있어서 호출부 try-catch가 의미없었음
} catch(Exception e) {
    log.error("파일 삭제 작업 중 오류 발생: " + e.getMessage());
}
```

호출부에서 try-catch로 감싸고 있었지만 `deleteFiles()` 내부에서 예외를 삼키고 있어서 실제로는 아무 의미가 없는 상태였습니다. `deleteFiles()`가 예외를 throw하도록 수정했습니다.

```java
} catch(Exception e) {
    log.error("파일 삭제 작업 중 오류 발생: " + e.getMessage());
    throw e;
}
```

## 3. 설계 결정: 작업 목적과 UX 기준으로 판단

물리 파일 삭제 실패 시 예외를 던지면 `@Transactional` 롤백이 발생해 사용자 입장에서 작업 전체가 실패합니다. 이게 맞는지 아닌지는 작업의 목적과 UX를 기준으로 판단했습니다.

**모든 케이스에서 예외를 삼키기로 결정:**

```java
// 프로필 이미지 수정 시 기존 파일 삭제 실패
try {
    fileUploader.deleteFiles(List.of(oldProfileDTO), subDir);
} catch (Exception e) {
    log.error("기존 프로필 이미지 물리저장소 삭제 실패: " + e.getMessage());
    // 수정 자체는 성공했으므로 삼킴
}

// 프로필 이미지 삭제 기능
try {
    fileUploader.deleteFiles(List.of(oldFileDTO), subDir);
} catch (Exception e) {
    log.error("프로필 이미지 물리저장소 삭제 실패: " + e.getMessage());
    // DB는 이미 null로 업데이트됨, 유령 파일은 감수
}

// 회원 탈퇴 시 프로필 이미지 삭제 실패
try {
    fileUploader.deleteFiles(List.of(profileDTO), subDir);
} catch (Exception e) {
    log.error("회원 탈퇴 중 프로필 이미지 물리저장소 삭제 실패: " + e.getMessage());
    // 탈퇴 자체는 성공시킴
}
```

유령 파일이 남는 불일치는 감수하되, 사용자 경험을 우선했습니다. 유령 파일은 추후 `@Scheduled`로 주기적으로 정리할 예정입니다.

## 4. 결론 및 회고

- **예외를 삼킬지 던질지는 작업의 목적과 UX를 기준으로 결정한다**: 물리 파일 삭제 실패가 사용자에게 직접적인 영향을 주는지 아닌지를 따져야 합니다.
- **유령 파일은 운영 레벨에서 해결 가능한 문제다**: 심각도가 낮은 불일치는 `@Scheduled`로 정리하는 전략이 현실적입니다.
- **호출부 코드가 의미있으려면 피호출부도 예외를 던져야 한다**: deleteFiles()가 예외를 삼키고 있어서 호출부 try-catch가 무의미했던 것처럼, 예외 전파 흐름을 전체적으로 파악해야 합니다.
